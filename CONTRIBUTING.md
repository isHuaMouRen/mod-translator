## 1. 提交信息标准化

此项目采用行业规范提交信息

### 1.1. 格式

单行:

```
<type>(<scope>?): <subject>
```

多行:

```
<type>(<scope>?): <subject>

<body>

<footer>?
```

*带 `?` 的为可选项; `:` 为英文冒号, 与 `<subject>` 之间应有一个空格*

`type` - 前缀, 表明提交的类型，在 `type` 后添加 `!` 表示破坏性更改:
 * `feat` - 新增功能
 * `imp` - 改善某处代码
 * `fix` - 修复BUG
 * `docs` - 只修改了文档
 * `style` - 对代码格式进行调整，美化。不影响执行逻辑
 * `ref` - 大规模重构(优化屎山但不影响代码执行效果)
 * `perf` - 优化性能的改动
 * `test` - 添加或修复测试
 * `build` - 影响构建系统和外部依赖的更改
 * `ci` - ci和脚本修改
 * `chore` - 不修改代码和测试的杂项更改
 * `revert` - 撤销某次更改
 * `remove` - 移除某些东西

`scope` - 可选，代表更改的位置或某个功能，比如更改了设置功能，那么此项应为 `settings` 此项全部字母小写，用 `-` 连接，例如 `file-reader`

`subject` - 简要描述更改内容，中英不限

`body` (多行) - 用于详细说明改动内容

`footer` (多行) - 可选，用于描述附加信息比如关联某个issue

### 1.2. 示例

如果你添加了下载模块，那么提交信息应该是这样的:

```
feat(download): 添加下载模块
```

为某处代码添加注释，空行，便于理解。那么应该为:

```
style: 为xxx添加注释
```

## 2. 版本号命名规则

此项目采用 **语义化版本控制 (*Semantic Versioning*，简称 *SemVer*)** 命名规则

## 2.1. 格式

```
[Major].[Minor].[Patch]-[Tag].[TagId]
```

* `Major` - 主版本号，当进行不向后兼容的破坏性更改或重大变更时递增，次版本号和修订号归零
* `Minor` - 次版本号，当添加了一些小功能或者一些可以向下兼容的功能时递增，修订号归零
* `Patch` - 修订号，当进行一些小更新或小bug修复时递增
* `Tag` - 附加信息，表示特殊版本
* `TagId` - 附加信息的编号

### 2.1.1. 已有的Tag

|Tag|描述|
|-|-|
| `alpha` |表示某个版本的alpha开发阶段|
| `beta` |表示某个版本的测试阶段，功能仍不完善|
| `rc (Release Candidate)` |表示某个版本的发布候选版本|
| `pre (Pre-Release)` |表示某个版本的预发布版本，功能已经基本稳定，很快就会发布正式版|
| `hotfix` |表示某个版本的热修复版本，出现了某些重大Bug时使用|

## 2.2. 示例

* `1.0.0-pre.2` 代表 `1.0.0` 版本的第二个预发布版，这意味着即将发布 `1.0.0` 正式版
* `1.0.0` 代表 `1.0.0` 的正式版本
* `2.1.1-hotfix.1` 代表 有两次破坏性更改一个小更新与一个修订版的版本的一个热修复版
